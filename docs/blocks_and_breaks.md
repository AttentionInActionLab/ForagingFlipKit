# Practice Blocks and Breaks

To improve usability and data quality, the ForagingFlipKit supports adding practice blocks before the main experiment, as well as short breaks to prevent participant fatigue. 

Both features can be easily controlled via the `practice` and `breaks` objects in your `window.config`.

---

## 1. Practice Blocks

Practice blocks use the exact same task structure and `patch_types` as your main blocks. Instead of defining entirely new trial types, you simply specify a practice sequence using your existing patch IDs (e.g., "A" and "B"). 

This ensures the practice phase perfectly represents the main task, but allows you to tag the resulting data differently for your analysis.

```javascript
  "practice": {
    "enabled": true,
    "sequences": [
      "AABB", // A short sequence to let participants practice both conditions
      "BBAA"
    ],
    "selection_method": "participant", // Matches the main blocks logic
    "log_data": true,
    "phase_label": "practice" // This label will be attached to the output data
  },
```

## 2. Breaks Between Blocks

Breaks are essential for longer experiments to help participants maintain performance. You can configure breaks to appear automatically between block transitions (e.g., when condition "A" switches to "B" in your sequence) or after a specific number of patches.

```javascript
"breaks": {
    "enabled": true,
    "show_between_blocks": true,  // Triggers a break when the sequence letter changes (e.g., A -> B)
    "after_every_n_patches": 0,   // Triggers after a fixed number of trials (set to 0 to disable)
    "duration_ms": null,          // Set to e.g., 30000 for a strict 30s break, or null for self-paced
    "message_html": "<div class='break-screen' style='text-align: center;'><font size='+4' face='Comic Sans MS' color='#FFFFFF'>Short break!<br><br>Press any key or click to continue.</font></div>"
  },
```

## 3. Data Logging (Phase Tracking)

When using Practice Blocks and Breaks, the resulting data logs will automatically distinguish between the different phases of the experiment. This saves you from having to guess which trial belonged to which block later on.

Note: The structure below is a preview of the output data generated by the kit during runtime. Do not copy this into your configuration file.

```javascript
{
  "eventType": "trial",
  "participantId": "0",
  "trialIndex": 1,
  "patchType": "A",
  "phase": "practice",          // Clearly identified as a practice trial
  "responseTime": 782,
  "pointsCollected": 15
}
```
